# DentVision V2 — Архитектурный ревью и аудит текущего состояния

**Version:** 1.0
**Status:** Review / Pre-implementation gate
**Связанные документы:** [`DENTVISION_V2_INTEGRATION_PLAN.md`](DENTVISION_V2_INTEGRATION_PLAN.md) · [`DENTVISION_V2_PLATFORM_EXTENSIONS_PLAN.md`](DENTVISION_V2_PLATFORM_EXTENSIONS_PLAN.md) · [`../BLUEPRINT.md`](../BLUEPRINT.md)

Задача документа: «обдумать всё, как оно будет работать, чтобы не было багов и криво работающих функций», сравнить план с **текущим** проектом. Все выводы получены из чтения кода и **фактического запуска** приложения (frontend :5000 + backend :3001 + PostgreSQL) при настройке окружения.

---

## 0. Главный вывод

**Направление верное.** AI-first ядро, модульный backend (Express + Prisma), единый Intent Engine, дизайн-система, guest-режим, React Query, а также сама последовательность планов (IAM → Ecosystem → Finance → Platform Extensions) — архитектурно корректны и слоисто согласованы.

**Но:** текущее ядро содержит несколько **конкретных багов и системных несогласованностей**, которые уже сейчас ломают ключевые сценарии (CRM недоступен из UI, AI-чат отвечает пусто, список клиник не грузится). Если начать строить 8 новых модулей поверх этого ядра **без этапа стабилизации**, каждый новый модуль унаследует и размножит те же дефекты.

**Рекомендация:** вставить перед Фазой 1 короткий **«Этап стабилизации» (Фаза 0.5)** — точечные фиксы ядра (≈10 пунктов, см. §4). Они дёшевы и разблокируют весь план.

## 1. Что уже сделано хорошо (не ломать при рефакторинге)

- **Модульность backend** (`src/modules/*`) с чистым разделением доменов.
- **Единый Intent Engine** (`modules/ai/core/*`) — правильная основа для AI-узлов Workflow и AI-отчётов.
- **Prisma-схема** покрывает CRM/Shop/School/AI; связи и индексы в целом адекватны.
- **Guest-режим** (`modules/guest`, `optionalAuth`) — рабочий (Shop/AI открываются анонимно; проверено).
- **React Query + Zustand**, дизайн-система, ленивые роуты — современный фронтенд-стек.
- **Rate limiting, helmet, CORS** — базовая безопасность на месте.

## 2. Недочёты текущего проекта (с доказательствами)

### 2.1 Критические баги (воспроизведены запуском)

| ID | Симптом | Причина / файл | Влияние |
|----|---------|----------------|---------|
| **B1** | После F5/перезагрузки пользователь разлогинивается; CRM недоступен | `restoreSession()` определён в `src/store/auth.store.ts`, но **не вызывается нигде** (проверено grep). Токен лежит в `localStorage`, но сессия не восстанавливается | любой reload выкидывает в guest → CRM/аналитика недоступны |
| **B2** | После логина «Мои клиники» пустые, хотя клиники есть | `login()` в `auth.store.ts` читает `result.memberships` (верхний уровень), а backend `/api/auth/login` кладёт членства в `user.memberships` без деталей клиники → `clinics: []` | нельзя войти в рабочее пространство штатным путём |
| **B3** | Вход в CRM (Расписание) падает с «Что-то пошло не так» | Несогласованный конверт: `GET /api/patients` возвращает `{ok, data:{data:[...], total}}` (пагинация), а `api.getPatients()` типизирован как `Patient[]` и UI делает `patients.map(...)` (`src/pages/crm/Schedule.tsx:133`, `Cashier.tsx:603`, `MedicalCard.tsx:129`). `.map` по объекту-обёртке → TypeError → ErrorBoundary | **root cause** краха CRM из UI |
| **B4** | AI-чат показывает пустые ответы ассистента | Двойная распаковка: `apiRequest` уже возвращает `data.data`, а `aiChat()` в `src/utils/api.ts` снова берёт `res.data` → `d = {}` → `reply = ''` | флагманская AI-функция «работает криво» в UI (бэкенд отвечает 200) |

> B3 напрямую объясняет краш, который наблюдался при попытке пройти сценарий «логин → войти в клинику → создать пациента» во время настройки окружения.

### 2.2 Безопасность и авторизация

| ID | Проблема | Файл | Риск |
|----|----------|------|------|
| **S1** | `GET /api/clinics` и `GET /api/clinics/:id` **без `authenticate`** → публично отдаются список клиник и **ФИО/специальности участников** | `modules/clinics/clinics.routes.ts` (нет middleware на GET) | утечка персональных данных, нарушение требований РК к перс. данным |
| **S2** | Доменные роуты проверяют только `req.user.clinicId` из токена, но **не роль/право**. Любой член клиники (ASSISTANT, STUDENT) может создавать/удалять пациентов | `modules/patients/*` и др. (нет `requireRole`/permission) | горизонтальная и вертикальная эскалация прав |

### 2.3 Данные и деньги

| ID | Проблема | Файл | Риск |
|----|----------|------|------|
| **D1** | Деньги хранятся как `Float` (`Invoice.amount`, `Product.price`, `TreatmentPlan.price`, `Order.total`, `LabOrder.price`) | `prisma/schema.prisma` | ошибки округления; **прямой конфликт** с планом Finance (там суммы в минорных единицах `BigInt`) |
| **D2** | Все `@id @default("")` — ID генерирует приложение (`uid()`), а не БД | `prisma/schema.prisma` + `lib/helpers.ts` | любой `create` без `id: uid()` вставит пустую строку → коллизия `unique` на второй записи |

### 2.4 Консистентность и «мнимые» функции

| ID | Проблема | Файл | Риск |
|----|----------|------|------|
| **C1** | Конверт ответа несогласован: `patients`/`clinics` — пагинированы (`data.data`), `shop/products` — плоский массив (`data:[]`). Фронт местами ждёт массив, местами объект | routes + `src/utils/api.ts` | тот же класс ошибок, что B3; сломается при любом новом потребителе |
| **C2** | Десятки функций в `src/utils/api.ts` — заглушки `Promise.resolve([])` / `{ok:true}` (labOrders, expenses, promotions, documents, serviceAccess, профиль, сертификаты, отзывы, некоторые shop/school) | `src/utils/api.ts` | функции «выглядят рабочими», но молча ничего не делают → ложное ощущение готовности |
| **C3** | Две реализации backend: legacy `server/` (JS) и активная `dentvision-backend/` (TS) | корень репозитория | дрейф контрактов, путаница, риск править «не тот» бэкенд |
| **C4** | Дублирование инвайтов: `POST /api/clinics/:id/invite` генерирует код, **не сохраняя** его (мёртвый путь), а рабочий путь — `POST /api/auth/invitations` (пишет `ClinicInvitation`) | `clinics.routes.ts` vs `auth.routes.ts` | «кривая» функция приглашений в зависимости от вызванного эндпоинта |

### 2.5 Процесс и качество

| ID | Проблема | Влияние |
|----|----------|---------|
| **Q1** | На backend **нет автотестов**; `npm run lint` = 2 ошибки + 637 предупреждений; `npm run typecheck` завершается с ошибками (`src/utils/formatters.ts`, `reminders.ts`) | регрессии не отлавливаются; нельзя ставить CI-гейт |

## 3. Как недочёты повлияют на новые модули (если не чинить сейчас)

| Недочёт | Какой новый модуль усугубит | Что делать до старта модуля |
|---------|-----------------------------|-----------------------------|
| C1 конверт ответа | **Developer Platform** — публичный API v1 закрепит хаос контракта наружу | стандартизировать конверт **до** проектирования OpenAPI/`/api/v1` |
| D1 Float-деньги | **Finance / Partner Program** — комиссии (5–15%, 70/30, 50/30/20) посчитают неверно | миграция `Float → BigInt` (тиын) **до** Finance Core |
| S2 отсутствие permission-check | **Ecosystem/Partner** — открытие ролей Supplier/Lecturer/Partner расширит дыру | внедрить `requirePermission` (IAM) **до** новых ролей |
| B1/B2 сессия и членства | **IAM `switch-context`** — мульти-контекст не заработает поверх сломанного логина/restore | починить restore+login **в рамках** Фазы 1 (IAM) |
| B4 двойная распаковка AI | **Workflow Studio (AI-узлы)** и **Data Intelligence (AI-отчёты)** будут «пустыми» | починить `aiChat`/клиент **до** AI-надстроек |
| C2 заглушки | **любой модуль**, зависящий от «уже существующих» функций (документы, промо, lab) | заменить заглушки на реальные вызовы или явные фиче-флаги |
| C3 два backend | все модули | зафиксировать `dentvision-backend` как единственный; `server/` — в архив |
| Q1 нет тестов/красный lint | все модули | ввести тест-раннер и CI-гейты до масштабирования |

Вывод: последовательность планов правильная, но **между «сейчас» и «Фазой 1» обязателен этап стабилизации** — иначе IAM/Finance/Ecosystem строятся на дефектном фундаменте.

## 4. Этап стабилизации (Фаза 0.5) — must-fix до больших модулей

Точечные, недорогие фиксы ядра. Каждый — с критерием готовности (DoD).

1. **Единый конверт ответа.** Ввести один формат (`{ ok, data, meta? }`), где списки всегда `data: T[]` + `meta.pagination`. Обновить клиент `apiRequest` под один контракт. *DoD:* `getPatients` возвращает `Patient[]`; CRM Расписание открывается без краша (снимает **B3, C1**).
2. **Починить `aiChat`.** Убрать двойную распаковку. *DoD:* ответ ассистента виден в чате (снимает **B4**).
3. **Восстановление сессии.** Вызывать `restoreSession()` при старте приложения (bootstrap в `Providers`/корне). *DoD:* после F5 пользователь остаётся авторизован (снимает **B1**).
4. **Членства при логине.** Возвращать membership+clinic в ответе `login` (или грузить через `getMe` сразу после). *DoD:* «Мои клиники» показывает клиники (снимает **B2**).
5. **Закрыть публичные clinics-эндпоинты.** Навесить `authenticate` (и позже `requirePermission`). *DoD:* `GET /api/clinics` без токена → 401 (снимает **S1**).
6. **Permission-checks на доменных роутах (deny-by-default).** Хотя бы грубо (роль) до полноценного IAM. *DoD:* ASSISTANT не может удалить пациента (частично снимает **S2**).
7. **ID на уровне БД.** Перевести `@id @default("")` → `@default(uuid())`. *DoD:* создание записи без явного `id` не падает (снимает **D2**).
8. **Деньги в минорных единицах.** Спланировать миграцию `Float→BigInt` (тиын) — обязательна до Finance. *DoD:* витрины показывают корректные суммы; готово к комиссиям (снимает **D1**).
9. **Убрать заглушки/дубли.** Пометить `Promise.resolve([])` фиче-флагами или реализовать; удалить мёртвый `clinics/:id/invite`; заархивировать `server/`. *DoD:* нет «молчащих» функций (снимает **C2, C3, C4**).
10. **CI-гейты.** Backend тест-раннер, `lint=0 ошибок`, `typecheck=0`. *DoD:* пайплайн красный при регрессии (снимает **Q1**).

## 5. Anti-bug гардрейлы (принципы, чтобы функции не работали «криво»)

- **Единый контракт API + типовой клиент.** Один конверт ответа; типы генерируются из OpenAPI; фронт не «угадывает» форму данных.
- **Контрактные тесты.** Для публичного API v1 — контрактные/снапшот-тесты OpenAPI (защита от повторения C1 наружу).
- **Deny-by-default авторизация.** Каждый доменный роут закрыт `requirePermission`; открытость — только явным исключением (guest-скоупы).
- **Единый источник прав — сервер.** Фронт лишь отражает `GET /api/iam/permissions`; никаких зашитых матриц прав как источника истины.
- **Деньги только в minor units + property-тест `debit == credit`** на каждую транзакцию; распределение остатка комиссии по правилу.
- **Идемпотентность** платежей и вебхуков (по `externalId`/ключу) — до Kaspi QR и Developer Platform.
- **Никаких «тихих заглушек» в проде.** Незавершённая функция = фиче-флаг + явная ошибка/заглушка UI, а не `Promise.resolve([])`.
- **Восстановление сессии и единый контекст** (IAM) — до открытия новых типов рабочих пространств.
- **Единый Action Registry** для AI/Workflow/API — одна реализация бизнес-действия, много точек вызова (исключает расхождение поведения).
- **Event-first** для сквозных эффектов (нотификации/аналитика/аудит/вебхуки) — вместо прямых вызовов между модулями.

## 6. Итоговая оценка

| Критерий | Оценка |
|----------|--------|
| Направление и слоистость архитектуры | ✅ верно |
| Последовательность планов (IAM→Ecosystem→Finance→Extensions) | ✅ корректна |
| Готовность **ядра** к масштабированию «как есть» | ⚠️ нет — нужен этап стабилизации |
| Основные риски | конверт ответа, деньги-Float, авторизация, восстановление сессии, заглушки |

**Резюме:** проект действительно идёт в правильном направлении, и планы V2 (IAM, Ecosystem, Finance, Developer Platform, Workflow Studio, Data Intelligence, Partner Program) архитектурно согласованы. Однако **до** их реализации необходимо закрыть ≈10 конкретных недочётов ядра (§4), иначе новые модули унаследуют текущие баги (пустой AI-чат, крах CRM, дыры авторизации, ошибки денег). Правильный порядок: **Стабилизация (Фаза 0.5) → Фаза 0 (Event Bus/Money) → Фаза 1 (IAM) → далее по планам.**
