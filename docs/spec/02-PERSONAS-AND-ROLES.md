# 02 — Personas, Roles & Jobs-to-be-Done

## 2.1 Primary Users (canonical)

Четыре основных пользователя продукта:

| # | Persona | Код | Суть |
|---|---------|-----|------|
| 1 | **Врач** | `doctor` | Клиническая работа, пациенты, планы, обучение |
| 2 | **Владелец** | `owner` | Бизнес, прибыль, контроль сети клиник |
| 3 | **Админ** | `admin` | Операционка, запись, касса, документы |
| 4 | **Покупатель** | `buyer` | Закупки в Marketplace (часто пересекается с owner/admin) |

`buyer` — продуктовая персона Shop. В системе это может быть та же учётка с ролью owner/admin/manager, либо отдельный закупочный доступ.

## 2.4 Permission Philosophy

1. AI видит только то, что разрешено роли.
2. Memory хранит больше, чем UI показывает; retrieval всегда через RBAC filter.
3. Клинические записи: doctor/owner (policy) — write; admin — operational write; others — least privilege.
4. Финансы: owner/cashier/admin по политике клиники.
5. Marketplace seller tools: только `supplier` (+ platform admin).

---

## 2.5 Anti-Personas

Не проектируем ядро продукта вокруг:

- Пациента как daily user платформы (пациент — public booking / portal later)
- Универсального «офисного менеджера без dental-контекста»
- Маркетплейс-туриста без связи с клиникой (вторичный трафик, не north star)

Пациентский опыт существует (запись, подпись документов), но **не является primary surface** V1–V3.
